Accessibility Evaluation for Library of Things

Library of Things is a London-based social enterprise that allows people to borrow products and tools such as for the house and garden, as well as musical instruments and various electronics with a growing range of selection via multiple branches. They value accessibility and inclusivity, therefore they wanted to have their website evaluated according to the industry standards based on WCAG 2.1 AA, the most current guidelines at the time of the evaluation; currently WCAG 3.0, which aims to add on WCAG 2.1 is under development.

After the preliminary review, 4 pages, as detailed below, were analysed initially with the automated tool WAVE plugin on Chrome simultaneously with the Inspect function. Automated testing was followed by manual testing, which involved the use of VoiceOver on MAC OSX 10.14 and Safari browser, as well as testing the quality of the navigation with the keyboard.

With the VoiceOver, the tests were as follows:

- Test the feature by reading every element.
- Test the feature by reading headers.
- Test the feature by tabbing through links.
- Test the feature by reading landmarks.
- Test the feature for use of ARIA.


The details of the accessibility barriers and errors based on WCAG 2.1 AA guidelines were documented in the main report.

* Due to confidentiality requirements, the main report is not to be published online. This page includes only selected examples of findings presented in the Appendices accompanying the main report, following a comprehensive accessibility evaluation carried out based on WCAG 2.1 AA.

1. Executive Summary

This report describes the conformance of the website of Library of Things with W3C’s Web Content Accessibility Guidelines 2.1 (WCAG 2.1). The review process is described in Section 5 below and is based on evaluation described in Accessibility Evaluation Resources.
Based on this evaluation, the Library of Things website does not meet the WCAG 2.1, Conformance Level AA due to various critical functional problems outlines in the report.

2. Background About Evaluation

Conformance evaluation of web accessibility requires a combination of semi-automated evaluation tools and manual evaluation by an experienced reviewer. The evaluation results in this report are based on evaluation conducted on the following date(s): 7 and 8th December, 2020. The website may have changed since that time.
I aimed to prioritise the most obvious accessibility barriers so that those which are likely to have the greatest impact in terms of functionality can be identified and addressed first.

Examples of Findings:

PAGE TITLE

The best practice is to have a short, clear and descriptive title so that the user can quickly understand the page content and decide whether it is the relevant page for them. For a visually impaired user, the above title may not indicate which page of the website they are on, unlike a user without a visual impairment, who may not need to check the title to know which page they are viewing.

Title: Library of Things | Borrow useful Things for your home, projects and adventures The page title is too long. The title is read by the screen reader and is very important for the visually impaired user as the first line of information.

The slogan “Borrow useful things for your home, projects and adventures” can be removed and the page title can be changed to “Home – Library of Things”
“Home – Library of Things” Lists the most specific information first + Identifies the site + Concise + Descriptive

Image text alternatives ("alt text") (pictures, illustrations, charts, etc.)
The aim of text alternatives is to provide information about an image to a user who is not able to see it due to a visual impairment. If they need to have alt text, meaningful and clear descriptions should be used. Otherwise, they should have null alt (alt="") to indicate that they are decorative.
The images do not contain “alt text” and are ignored by the screen reader. They have null alt (alt="") indicating that they are decorative images without any function and can safely be ignored.

TEXT

Headings

Headings need to be marked up to indicate the page’s content structure to the user so that the assistive technologies can identify them and navigate the page effectively.

Findings:
a-The page is missing the first level heading. ‘Why buy when you can borrow?’, which is currently within a paragraph, is logically the most important title since it introduces the website.

Recommendation: Use < h1 > element for the most important title ‘Why buy when you can borrow?’

b-A logical hierarchy would suggest that the title ‘Our most popular things’ should have a h2 element, which is currently using a h3 (< h3 >) element

Similarly, “We are delivering to Brixton too” and “News Flash we are coming to Morden”, are using h3 elements (< h3 >), which should be moved up to the h2 level.

Recommendation: Use < h2 > element for the titles ‘Our most popular things’, “We are delivering to Brixton too” and “News Flash we are coming to Morden”, to follow a logical hierarchy.


Contrast Ratio (colour contrast)

Guidance
The visual presentation of text and images of text should have a contrast ratio of at least 4.5:1.

Large Text: Large-scale text and images of large-scale text have a contrast ratio of at least 3:1.
Incidental: Text or images of text that are part of an inactive user interface component, that are pure decoration, that are not visible to anyone, or that are part of a picture that contains significant other visual content, have no contrast requirement.
Logotypes: Text that is part of a logo or brand name has no contrast requirement.

For: People who cannot use a mouse and use only the keyboard, and people who use a screen reader.

Findings: 8 contrast errors, all in the same section “Our most popular things”.

The pricing information for all the products against the white background lacks contrast. The lack of contrast is noticeable even for a user with no visual impairment. When the contrast is checked in more detail with the Colour Contrast Analyser, three tones of grey colour of the foreground texts are revealed; two of which meet the contrast requirements of WCAG 2.1 AA, however as a whole, the contrast fails.
It is possible to see the colour in great detail with the Colour Contrast Analyser’s advanced zoom function.

It also allows us to assess whether the colour will be suitable for users with different types of colour blindness condition.
As it is seen in the first image below on the left, the light grey will be problematic not only because of lack of contrast, but also for the colour blind to accurately see the colour of the content.

#D4D2D1(foreground) on #FEFFFE (background) fails all of the contrast requirements of WCAG 2.1AA

#7A7473 (foreground) on #FEFFFE (background) fails the contrast requirements of WCAG 2.1AAA, however, satisfies AA the with a contrast ratio of 4.6:1

Recommendation:

The contrast between the (foreground) text colour and the background colour should be increased. By taking the above finding as reference, #7A7473 (AA) or #544C4B (AAA) could be used against the current background colour.


Resize Text (Level AA)

Requirement: Except for captions and images of text, text can be resized without assistive technology up to 200 percent without loss of content or functionality. (WCAG 1.4.4)

For: People with low vision, low-contrast vision, or other imperfect vision problems

Findings: No problems found with resizing. It was checked by zooming in 200% from Chrome settings. Images and the text appeared clear. People with low vision, low-contrast vision, or otherwise less-than-perfect vision may wish to magnify web content in order to make it legible or less-strenuous to read.


Interaction

Keyboard access and visual focus

Findings:

Tab skips drop-down menus “About” and “Locations” in the global navigation. Recommendation: Make the drop-down menu items selectable so that it is not missed and the main menu item, as well as the dropdown menu items are reachable; otherwise, the user won’t be able to access these sections.

During tabbing, “Start Borrowing” button misses a visual indicator to inform the user that it is selected.

Recommendation:

This may not cause a problem for a screen reader user, however, someone with low-vision or even no vision problems, would not be able to see that the button is in focus.

The heading structure is not logical and is preventing the user from efficient navigation.

Recommendation:
The heading structure recommendations above, starting from the use of h1 element and adding the relevant elements to the other headings would solve the issue.

The images except for the product images can not be tabbed due to not being linked to any section.

Recommendation:

Link all the images to relevant sections/content for logical navigation.

Basic Structure Check

Performed with Web Dev tool by disabling all images, disabling all styles (CSS) and linearising the page.

Findings:

Navigation problems due to the previously mentioned problems with the 2 global navigation drop down menu items and the lack of logical heading structure. A new finding with the Web Dev tool is that the global navigation menu appears to have a duplicate. The first uses links for Sign In/Sign Up (< a href >), and the second has a button element for the same items.
